## API Key Management

API key definitions live in `API_KEYS_CONFIG` in
`backend/app/api/settings/api_keys.py`. In Supabase mode, the Settings -> API
Keys UI stores encrypted values per workspace in `workspace_provider_secrets`.
`backend/.env` is the bootstrap/default fallback for local development and
non-Supabase mode.

**Categories:** `ai` (Anthropic, OpenAI, ElevenLabs, Gemini, Nano Banana, VEO),
`storage` (Pinecone), `utility` (Tavily, Google OAuth, Webshare),
`integrations` (Notion, Jira, Freshdesk, Mixpanel, Opik).

**Adding a new API key:**
1. Add entry to `API_KEYS_CONFIG` with `id`, `name`, `description`, `category`
2. Create validator beside the owning provider or connector (make minimal API call to verify)
3. Add validation routing in `_validate_key(...)` in `backend/app/api/settings/api_keys.py`
4. Add any cached-client reload hook in `update_api_keys()` so workspace-secret updates take effect without restart
5. Frontend auto-renders from config — only update `ApiKeysSection.tsx` if adding a new category to the type union

**Key patterns:**
- Keys are masked before sending to frontend (`EnvService.mask_key()`)
- Masked values (starting with `***`) are skipped on save
- Validation tests real API connections before saving
- Provider clients and connectors must resolve workspace secret first, then environment fallback
- Services with cached config (Notion, Jira, Freshdesk, Mixpanel, Opik/OpenAI/Anthropic clients) need reload hooks in `update_api_keys()`

## Source Processing Pipeline

Status flow: `uploaded → processing → ready`, with embedding behavior owned by
the source type. Document-style sources generally chunk/embed processed content
for hybrid search. CSV sources are intentionally analyzed on demand and are not
embedded; connector/analysis sources may store schema or summary snapshots plus
tool-specific metadata.

### Processed Output Format

Document-style processors use `build_processed_output()` from
`app.sources.text.processed_output` for consistent output format:

```
# Extracted from {TYPE} document: {source_name}
# Type: {TYPE}
# Total pages: {N}
# Processed at: {timestamp}
# {metadata_key}: {value}
# ...
# token_count: {count or "200k+"}
# ---

=== {TYPE} PAGE 1 of N ===

{page content}

=== {TYPE} PAGE 2 of N ===

{page content}
```

**Key Design Decisions:**
- **Sources with logical pages** (PDF, PPTX, Image batch): Preserve real page/slide structure
- **Sources without logical pages** (TEXT, DOCX, AUDIO, LINK, YOUTUBE, RESEARCH): Single page marker `=== TYPE PAGE 1 of 1 ===`
- **Analysis/connector sources** (CSV, DATABASE, FRESHDESK, JIRA, MIXPANEL, MCP): Domain-specific processed snapshots or metadata; do not assume page markers
- **Token-based chunking** handles splitting when embeddings are produced (~200 tokens per chunk)
- Each page extraction is **self-contained** with context from surrounding pages included

### Page Markers
Standard processed document content uses page markers: `=== {TYPE} PAGE 1 of N ===`
Types include PDF, TEXT, DOCX, PPTX, AUDIO, IMAGE, LINK, YOUTUBE, and RESEARCH.
Analysis/connector sources may use their own processed text format.

### Token Counting (Hybrid Approach)

Token counting uses **tiktoken** (local) for speed, with Anthropic token-counting available for exact Claude counts when needed.

**Why tiktoken?** Chunking calls `count_tokens()` thousands of times (per page, per sentence, per word for long sentences). API calls would take minutes due to network latency. tiktoken is local and instant.

```python
# app.sources.tokens / app.providers.anthropic.token_count
count_tokens(text)      # Uses tiktoken (fast, local) - for chunking operations
count_tokens_api(text)  # Uses provider token counting when exact counts are needed
```

tiktoken uses `cl100k_base` encoding for fast approximate chunking. Use provider
token-counting only when exact billing/quota behavior needs it.

## Web Agent Architecture

**Agentic loop pattern** for URL content extraction. Loop: provider model calls tools → execute → return result → repeat until termination.

**Tool Types**:
- **Provider/server tools**: Provider handles execution when supported (for example web fetch/search)
- **Client tools**: We execute through the source link agent modules, including `sources/link/agent.py` and `sources/link/run.py`
- **Termination Tool**: `return_search_result` signals completion

Execution logs saved to `data/projects/{id}/agents/web_agent/{execution_id}.json` for debugging.

## Main Chat Architecture

**RAG agentic loop** for source-aware conversations. System prompt includes dynamic source context + memory via `app.config.context.context_loader`.

### Tool Use Loop
```

    ↓

          → No:  Store final text → Return to user
```

### Source Search (Hybrid Search)

The `search_sources` tool uses a smart hybrid search strategy based on source size:

**Tool Contract** (`backend/app/chat/tools/specs.py`):
`search_sources` is a typed `ToolSpec` with a Pydantic input model containing
`source_id`, optional short `keywords`, and optional semantic `query`. Provider
adapters compile that single contract to Anthropic/OpenAI tool schemas.

**Search Strategy** (`app.sources.search.SourceSearchExecutor`):
```
if source.token_count < 1000:
    → Return ALL chunks (no search needed)
else:
    → Local keyword search (fuzzy matching via difflib)
    → Semantic search (OpenAI embedding → Pinecone)
    → Combine & dedupe by chunk_id
```

**Key Point**: The model passes `source_id` to search, but receives `chunk_id` in results. Citations must use `chunk_id`, NOT `source_id`.

### Chunk-Based Citations

Citations use chunk_ids for precise references to specific content sections.

**Citation Format**: `[[cite:CHUNK_ID]]`
**Chunk ID Format**: `{source_id}_page_{page}_chunk_{n}`
**Example**: `[[cite:abc123-def456_page_5_chunk_2]]`

**API Endpoint**: `GET /projects/{id}/citations/{chunk_id}`
Returns chunk content for tooltip/popover display.

**Flow**:
```
Model response: "Information here [[cite:abc123_page_5_chunk_2]]"
       ↓
Frontend parses → extracts chunk_id
       ↓
Hover → GET /api/v1/projects/{id}/citations/{chunk_id}
       ↓
Backend loads chunk from Supabase → returns content + metadata
       ↓
Tooltip shows: "Source Name - Page 5, Section 2" + content
```

### Debug Logging
Debug logging for API calls is available during development for troubleshooting chat interactions.

## Provider Response Parsing

Provider parsing is adapter-owned. Domain code should consume typed runtime
results and events, not Anthropic content blocks directly:

```
domain code → app.agents.runtime.RunRequest
         ↓
ProviderAdapter (Anthropic Messages or OpenAI Responses)
   - compiles domain ToolSpec contracts to provider schemas
   - translates provider protocol items into RunResult / RunEvent
   - preserves provider-specific ordering rules at the edge
         ↓
domain code stores neutral content parts or executes typed tool calls
```

**Tool use flow**: provider adapter returns `RunResult(status="requires_tools")`
with `ToolCall`s → runtime/domain executor validates inputs through `ToolSpec`
→ every call gets a `ToolResult` → provider adapter formats the next provider
turn.

## Cost Tracking

Per-project API cost tracking. Pricing is provider/model-aware. Model calls
pass `project_id` through the runtime/provider adapter boundary so usage can be
stored in Supabase `projects.costs` and displayed in ProjectHeader with tooltip
breakdown.

## Memory System

**Tool-based memory** for persistent context across conversations.

- **User Memory** (Supabase `users.memory` column): Global preferences across all projects
- **Project Memory** (Supabase `projects.memory` column): Project-specific context

**Flow**: The model calls `store_memory` → returns immediately (non-blocking)
→ background task uses the configured memory model to merge new + existing
memory (max 150 tokens) → saved to Supabase → included in future system prompts
via `app.config.context.context_loader`.

## Tier Configuration

Centralized provider/rate configuration lives in `app/config/provider.py`. Set
provider tiers with `ANTHROPIC_TIER`, `OPENAI_TIER`, and `PINECONE_TIER` in
workspace secrets or environment fallback (1-4).

| Tier | Workers | Pages/min | Use Case |
|------|---------|-----------|----------|
| 1 | 4 | 10 | Free tier |
| 2 | 16 | 100 | Standard |
| 3 | 24 | 200 | Pro |
| 4 | 80 | 1500 | Enterprise/Demos |

**Tier 4 Optimization**: Workers can be high (80) because PDF/PPTX processing is I/O-bound (waiting for API), not CPU-bound. The 4000 RPM limit with 5-page batches theoretically supports ~20,000 pages/min, but output token limits (~800K/min) cap practical throughput.

## AI Integration Standard Pattern

NBB-011 moved NoobBook from direct Anthropic-shaped callers to a typed
provider-neutral runtime at `app.agents.runtime`. New model/tool orchestration
code must target that runtime boundary instead of importing raw Claude service
methods, Anthropic response parsers, or static tool JSON from domain code.

New tools are domain-owned Pydantic `ToolSpec`s; provider adapters compile
those specs to Anthropic/OpenAI schemas. Prompt JSON remains content/config
data, but its loaded shape is validated through typed prompt specs.

New integrations must live in domain-owned homes such as `sources/`, `chat/`,
`studio/`, `providers/`, or `connectors/`, not former mechanism buckets under
`services/`.

### Required Steps (Mandatory)

```
1. CONFIGURATION LOADING
   ├── typed prompt spec                                # System prompt, provider, model, temperature, max tokens
   ├── domain-owned ToolSpec                            # Pydantic input/output + handler
   └── provider config                                  # Tier/rate limits by provider

2. PATH MANAGEMENT
   └── app.base.paths.get_*_dir(project_id)             # Use base path helpers for local runtime dirs
       ├── get_processed_dir()     # For output files
       ├── get_raw_dir()           # For input files
       ├── get_chunks_dir()        # For chunked text
       └── get_web_agent_dir()     # For debug agent logs

3. RUNTIME CALL
   └── RuntimeRunner + ProviderAdapter                  # Emits typed RunResult / RunEvent values
       ├── validates ToolSpec input before handlers run
       ├── pairs every tool call with a tool result
       ├── records usage and provider metadata
       └── keeps provider protocol conversion in providers/*
```

### Optional Steps (As Needed)

```
FOR BATCHED PROCESSING (PDF, PPTX):
├── app.sources.extract.batching.create_batches(items, DEFAULT_BATCH_SIZE)
├── app.sources.extract.batching.get_batch_info(items, batch_size)
└── DEFAULT_BATCH_SIZE = 5                              # Standard batch size

FOR RATE-LIMITED APIs:
├── app.providers.anthropic.rate.RateLimiter(requests_per_minute)
└── rate_limiter.wait_if_needed()                       # Call before each API request

FOR PARALLEL PROCESSING:
├── ThreadPoolExecutor(max_workers=tier_config["max_workers"])
├── task_service.is_target_cancelled(source_id)         # Check for user cancellation
└── Cooperative cancellation pattern                    # Raise CancelledException

FOR BINARY DATA:
└── app.providers.anthropic.media.encode_bytes_to_base64(data)  # Base64 encoding for Anthropic media payloads

FOR FILE-TYPE SPECIFIC:
├── app.sources.pdf.ops.get_page_count(), get_all_page_bytes()    # PDF operations
├── app.sources.docx.ops.extract_text_from_docx()                 # DOCX extraction
└── app.sources.pptx.ops.convert_pptx_to_pdf()                    # PPTX to PDF conversion
```

### Runtime Template

```python
from app.config.prompt import prompt_loader
from app.agents.runtime import (
    RunLimits,
    RunMessage,
    RunRequest,
    TextPart,
    run_with_provider,
)
from app.chat.tools.specs import source_search_tool


def answer(project_id: str, user_text: str) -> str:
    prompt = prompt_loader.get_prompt_config("default")
    result = run_with_provider(
        RunRequest(
            provider=prompt["provider"],
            model=prompt["model"],
            purpose="chat",
            system_prompt=prompt["system_prompt"],
            messages=[RunMessage(role="user", content=[TextPart(text=user_text)])],
            tools=[source_search_tool()],
            limits=RunLimits(
                max_output_tokens=prompt["max_tokens"],
                temperature=prompt["temperature"],
            ),
            project_id=project_id,
        )
    )
    return result.text
```

### What NOT to Do

- **Never** duplicate configuration loading logic - use typed prompt specs and provider config helpers
- **Never** hardcode paths - use `app.base.paths` functions
- **Never** parse provider responses in domains - provider adapters translate to runtime contracts
- **Never** implement custom rate limiting - use `RateLimiter` class
- **Never** implement custom batching - use `create_batches()` utility
- **Never** skip `project_id` in API calls - needed for cost tracking
