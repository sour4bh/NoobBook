## Goal
Create the studio taxonomy before moving individual items.

## Current repo cues
- `backend/app/api/studio/`
- `backend/app/services/studio_services/`
- `backend/app/services/tool_executors/`
- `backend/app/services/tools/`

## Scope
- Create `backend/app/studio/` with category roots such as `documents/`, `marketing/`, `design/`, `learning/`, `media/`, and `shared/`.
- Define the naming map from current studio item names to category/item directories.
- Create small public surfaces and package docs for studio categories.
- Keep shared concerns minimal and explicit; only add `studio/shared/` for real shared logic such as registry, shared schemas, or common artifact helpers.
- Add temporary registry or import shims to support staged migration of individual items.

## Out of scope
- Migrating all studio items in one ticket.
- Rewriting studio logic.
- Changing studio route contracts.

## Deliverables
- Studio category skeleton.
- Old-to-new naming map.
- Minimal shared surfaces for staged migration.

## Acceptance criteria
- The target studio taxonomy exists.
- Contributors can tell where a new studio item belongs from the directory structure alone.
- No category is forced into a generic `studio_services` or `tool_executors` bucket.

## Notes
Do not over-abstract `studio/shared/`. It should hold actual shared logic, not become the new dumping ground.
