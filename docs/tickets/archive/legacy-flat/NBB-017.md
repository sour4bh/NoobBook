## Goal
Let `lib/` and `contexts/` stay conventional, but only as truthful boundary folders.

## Current repo cues
- `frontend/src/lib/api/`
- `frontend/src/lib/auth/`
- `frontend/src/lib/citations.ts`
- `frontend/src/lib/exportChatMarkdown.ts`
- `frontend/src/lib/exportChatPdf.ts`
- `frontend/src/lib/logger.ts`
- `frontend/src/lib/utils.ts`
- `frontend/src/contexts/PermissionsContext.tsx`

## Scope
- Keep `lib/api/` and `lib/auth/` as browser/platform boundary folders.
- Review `citations.ts`, `exportChatMarkdown.ts`, `exportChatPdf.ts`, and `utils.ts` for ownership; move product-specific helpers to their owning domains where appropriate.
- Leave `logger.ts` in `lib/` unless it grows product semantics.
- Keep `contexts/` for app-wide providers only; if more global providers appear, group them intentionally (for example `contexts/app/`).
- Do not pass browser/framework objects deep into domain logic when plain values will do.

## Out of scope
- Introducing a new frontend architecture taxonomy.
- Large UI behavior rewrites.
- Moving every helper out of `lib/` for ideological reasons.

## Deliverables
- Cleaner `lib/` with explicit boundary ownership.
- Small, clearly app-wide `contexts/` directory.
- Updated imports from components and hooks.

## Acceptance criteria
- `lib/` contains adapters and tiny neutral helpers, not product orchestration.
- `contexts/` remains small and clearly app-wide.
- Any file in `lib/` that speaks in product nouns has been reviewed and either justified or moved.

## Notes
This is about truthfulness, not purity. If a file really is a boundary adapter, `lib/` is the right home.
