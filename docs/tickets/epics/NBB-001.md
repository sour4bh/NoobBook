# NBB-001 - Foundation and Migration Guardrails

## Goal

Stop new structural drift, reconcile stale repository guidance, and put a real safety net in place before domain migrations begin.

## Outcome

Contributors can find the active repo/branch guidance, new code has approved destinations, CI exists, and baseline route/auth tests catch obvious breakage before the migration moves core behavior.

### NBB-103 - Bootstrap GitHub Actions CI and add early no-new-legacy-files guardrail

**Goal:** Create a CI lane and block new files in frozen legacy buckets before migration starts.

**Primary write scope:**
- `.github/workflows/`
- structural guardrail script path chosen by this ticket
- repo-root structure/contributor docs that name CI checks
- `.mcp.json.example`
- `.gitignore`
- `CONTRIBUTING.md` (one link only, pointing at the new setup doc)
- `docs/tickets/REFACTORY_SETUP.md`
- `docs/tickets/move-plan.csv`
- `docs/tickets/helpers/json_asset_move.sh`
- `docs/tickets/helpers/string_ref_scan.py`
- `docs/tickets/README.md`

**Current repo cues:**
- There is no `.github/workflows/` directory.
- Current tickets assumed CI would exist.

**Scope:**
- Use GitHub Actions as the CI platform.
- Add a minimal workflow that runs backend tests and the structural guardrail.
- Add a structural check that fails on new files under frozen destinations except allowlisted temporary shims.
- Frozen destinations must include `backend/app/services/`, `backend/app/utils/`, `backend/app/services/ai_agents/`, `backend/app/services/ai_services/`, `backend/app/services/tool_executors/`, `backend/app/services/tools/`, `backend/app/services/studio_services/jobs/`, `backend/app/services/studio_services/studio_processing/`, `backend/data/prompts/`, and `frontend/src/components/hooks/`.
- Wire the refactory MCP plugin (`~/dev/refactory` or equivalent) into NoobBook tooling so movement tickets share one codemod backend. Refactory exposes `move_module`, `move_symbol`, `rename_symbol`, and `validate_imports` backed by rope (Python) and ts-morph (TypeScript). Movement tickets call refactory tools directly during their turn; every call is dry-run-first.
- Land `.mcp.json.example` at the NoobBook root documenting the refactory MCP server entry. `.mcp.json` itself is added to `.gitignore` — each contributor copies the template and edits the path to their local refactory checkout. Absolute paths are not committed to the repo.
- Land `docs/tickets/REFACTORY_SETUP.md` naming the refactory plugin-dir setup as the recommended path (auto-runs refactory's `SessionStart` hook that installs `rope`, `mcp`, and `ts-morph`) and the raw-`.mcp.json` fallback path (requires manual dependency install because `SessionStart` is skipped). Include the agent self-check rule: if `mcp__refactory__move_module` does not appear in `tool_search`, the plugin is not loaded; stop and fix session setup before editing code. Keep `CONTRIBUTING.md` untouched except for one link to `REFACTORY_SETUP.md`; branch-guidance reconciliation remains `NBB-101`'s scope.
- Land `docs/tickets/move-plan.csv` as an append-only bookkeeping log with header row only at this point; movement tickets append their own rows. Columns (no spaces, standard CSV): `ticket,language,old_path,new_path,old_symbol,new_symbol,mode,tool`. Modes: `python_module_move`, `python_symbol_rename`, `python_symbol_move`, `json_asset_move`, `text_reference_check`. Tool values: `refactory:move_module` | `refactory:rename_symbol` | `refactory:move_symbol` | `json_asset_move.sh` | `string_ref_scan.py` | `manual`.
- Land `docs/tickets/helpers/json_asset_move.sh` — `git mv` plus flags loader-registry references for prompt/tool JSON moves (NBB-207B/C). The helper does not edit registry files; an agent updates any flagged entries by hand after the move lands. Refactory does not handle JSON assets.
- Land `docs/tickets/helpers/string_ref_scan.py` — grep over docs, tests, and configs for string module references (monkeypatch targets, `importlib` strings, `CLAUDE.md`/`AGENTS.md`/ticket bodies). `validate_imports` is import-statement only, so string refs need a separate scan.
- `move-plan.csv` is read-only metadata, not a driver script. Its purpose is consolidated audit ("where did `utils/pdf_utils.py` land?") and gating (a future `dag.py` check can verify every mover ticket produced at least one row).
- Document the agent execution convention for movement tickets:
  1. Confirm refactory MCP tools are available (`mcp__refactory__move_module` etc. appear in `tool_search`). If not, the plugin is not loaded — stop and fix session setup before editing code.
  2. For `move_symbol` rows only: create an empty target file first (`touch backend/app/<new_path>`) — refactory's Python `move_symbol` dry-run requires the target module to exist on disk.
  3. Call the refactory tool with `dry_run=true`, review the diff.
  4. Call the refactory tool with `dry_run=false` to apply. Refactory's MCP schema defaults `dry_run` to `false`; the dry-run-first rule is agent policy, not tool-enforced.
  5. Append one row per move to `move-plan.csv` with the ticket id.
  6. Run `string_ref_scan.py` for the old path and resolve any hits.
  7. Run `pyright` on touched packages and the `NBB-106` route smokes plus `NBB-107` auth tests. These post-apply checks are NoobBook's, not refactory's; the ticket is not done until they pass.

**Out of scope:**
- Rich import-boundary enforcement; that belongs to `NBB-704A` and `NBB-704B`.
- Full frontend CI unless `NBB-108A` chooses to add it through `NBB-108B`.
- Extending refactory itself. JSON assets, string-reference scans, and parameter-level codemods stay in NoobBook helpers or remain manual.

**Acceptance criteria:**
- CI exists in GitHub Actions.
- CI fails when new non-allowlisted files are added to frozen legacy buckets.
- The allowlist is explicit and reviewed.
- `.mcp.json.example` exists at the repo root; `.gitignore` excludes `.mcp.json`.
- `docs/tickets/REFACTORY_SETUP.md` exists, documents both the `--plugin-dir` path (recommended; auto-installs dependencies via `SessionStart`) and the raw-`.mcp.json` fallback (`uv run --with rope --with mcp` for Python dependencies, plus manual `pnpm install` in `tsmorph/`), and carries the "if `tool_search` doesn't show refactory, the plugin isn't loaded" warning.
- `CONTRIBUTING.md` is untouched beyond a single link pointing agents at `REFACTORY_SETUP.md`.
- A smoke test confirms `mcp__refactory__move_module` is callable with `dry_run=true` on a throwaway file from a session with the plugin enabled.
- `docs/tickets/move-plan.csv` exists with header row only and parses with `csv.DictReader` (no spaces in header).
- `docs/tickets/helpers/json_asset_move.sh` exists, defaults to dry-run, requires explicit `--apply` to mutate, and runs `--help` cleanly.
- `docs/tickets/helpers/string_ref_scan.py` exists, covers Python and TypeScript/JavaScript extensions (`.py`, `.ts`, `.tsx`, `.js`, `.jsx`, `.mjs`, `.cjs` plus docs/configs), and runs on an empty pattern without error.
- `docs/tickets/README.md` has a "Move bookkeeping" subsection pointing at `move-plan.csv`, the helpers, the dry-run-first agent convention, and the post-apply checklist.
- `NBB-103` is the only ticket that creates these files; movement tickets only append rows or call the helpers.

**Verification:**
- Run the structural check locally.
- Confirm GitHub Actions workflow calls the check.
- Confirm `git check-ignore .mcp.json` reports the path is ignored.
- With refactory enabled, call `mcp__refactory__move_module` with `dry_run=true` on a throwaway target path; confirm it returns a diff without errors.

**Traceability:** Covers missing CI and guardrail timing findings, plus the migration-move tooling gap surfaced during decision-lock review.

### NBB-104 - Create backend skeleton, charters, dependency graph, and route-adapter policy

**Goal:** Create backend destinations and define package rules before moving behavior.

**Primary write scope:**
- `backend/app/`
- backend package README/docstring files
- repo-root structure guide

**Scope:**
- Create backend package skeletons for the approved domain roots.
- Define a dependency graph: route adapters call domain public surfaces; domains may depend on connector public surfaces; connectors may depend on provider clients; domains may depend on providers directly only for provider-neutral runtime primitives; domains must not reach into each other's internals.
- Define `backend/app/api` as transport-only for this program.
- Apply the `base/` and `shared/` charters below. No later ticket may create `base/` or `shared/` content without linking back to this charter.

**Decision map:**

Approved backend roots. No new backend root may be created without amending this list.

| Root | Purpose |
|---|---|
| `api/` | Transport adapters only: parse HTTP, run guards, call domain public surfaces, format responses. |
| `auth/` | Identity, route/service guards, permission policy, user identity store. |
| `projects/` | Project domain behavior and project store. |
| `chat/` | Chat loop, streaming, memory invocation, chat persistence coordination. |
| `sources/` | Source ingestion, search, citation, extraction, indexing, and analysis. |
| `studio/` | Studio generation items and studio job/tool/run behavior. |
| `brand/` | Brand config and brand asset ownership. |
| `settings/` | App/user settings and API-key UI backend. |
| `connectors/` | Product-level configured external capabilities, user/project connection stores, permission-gated tool surfaces. |
| `providers/` | Low-level external API clients, SDK wrappers, auth primitives, storage adapters, and runtime IO adapters. |
| `background/` | Task lifecycle, cancellation, active-task status, and execution-log ownership. |
| `base/` | Empty unless a charter exception proves 3+ domain consumers and no better owner. |

Legacy roots such as `utils/` and `services/` are not approved destinations. They are frozen by `NBB-103` and drained or reduced to approved exceptions by `NBB-705A` through `NBB-705E`.

| Directory family | Charter |
|---|---|
| `base/` | Reserved for truly cross-domain primitives with 3+ domain consumers and no better owner. `base/` ships empty at the end of Epic 001. No file lands in `base/` without a PR note explaining why no domain owns it. |
| `<domain>/shared/` | Allowed only after that domain has 3+ concrete slices using the same behavior. The shared code must name the boundary it serves, for example `studio/shared/` or `sources/analysis/shared/`. |
| Preemptive `shared/` directories | Forbidden. A directory cannot exist only to reserve a future convenience bucket. |
| Generic helpers in `base/` or `shared/` | Forbidden. Rehome to the owning domain, provider/client boundary, connector capability, or the approved `utils/` exceptions from `NBB-705E`. |

**Out of scope:**
- Moving route modules into domains; route movement is deferred.
- Large behavior refactors.

**Acceptance criteria:**
- Every new backend root has a stated owner and allowed import direction.
- `base/` is empty unless a PR proves 3+ domain consumers and no better owner.
- `shared/` exists only where 3+ concrete slices already share behavior.
- Route files are documented as transport adapters only.

**Verification:**
- Review package docs and dependency graph.
- `rg "shared/|base/" docs/tickets/epics docs/tickets/README.md`

**Traceability:** Covers `base/`/`shared/` loopholes, route policy, and dependency-direction gaps.

### NBB-109 - Apply pre-migration correctness sweep from the taste audit

**Goal:** Fix cheap correctness defects before structural churn moves the files and hides the blame.

**Primary write scope:**
- `backend/app/services/`
- `backend/app/api/chats/routes.py`
- `backend/app/api/messages/routes.py`
- `backend/app/utils/path_utils.py`
- files named by the taste-audit Phase 1 sweep

**Scope:**
- Fix the 16 `send_message` call sites missing explicit `project_id=` so cost tracking remains complete. Known audit offenders include `services/ai_services/pdf_service.py`, `services/ai_services/pptx_service.py`, `services/ai_agents/web_agent_service.py`, `services/ai_agents/email_agent_service.py`, other `ai_agents/*_agent_service.py` files, and some `tool_executors/*`; the verifier below enumerates the full set.
- Build `backend/scripts/verify_project_id_coverage.py`: an AST verifier over every call resolving to `claude_service.send_message`, including aliased imports and wrappers that re-expose `send_message` through `**kwargs`. It must understand `main_chat_service._call_claude` as a legitimate wrapper only when `_call_claude` itself receives/forwards `project_id`. It exits non-zero on any omission.
- Fix the N+1 query in `data_services/chat_service.py:list_chats()` before the chat store move in `NBB-209A`: replace the per-chat `SELECT count(*)` pattern with one Postgres join or subquery so chat listing issues one query regardless of chat count.
- Replace bare `except Exception:` swallowing only at `backend/app/api/chats/routes.py:103` and `backend/app/api/messages/routes.py:103` with narrow handling or `logger.exception(...)` plus explicit response behavior. Do not touch the optional Opik init broad catch in `backend/app/services/integrations/claude/claude_service.py`; that catch is a legitimate optional-dependency boundary.
- Replace the 14 hardcoded `Path()` filesystem accesses identified by the audit with `path_utils` where existing helpers apply. Worst offenders: `services/tool_executors/presentation_agent_executor.py` (4x), `services/tool_executors/deep_research_executor.py`, and `services/ai_services/pptx_service.py`.
- Remove the 17 `from __future__ import annotations` instances in `backend/app/`, including the audited `database`, `mcp`, and `rbac` modules.
- Add `backend/tests/test_claude_cost_tracking.py` or an equivalent focused test asserting that a `claude_service.send_message(...)` call without `project_id=` raises or logs a warning.

Note: these audited fixes are too small-grained for the refactory codemod engine (adding `project_id=` at 16 `send_message` call sites, removing individual `from __future__ import annotations` imports, narrowing bare excepts). They land as direct edits within this ticket and are not tracked in `docs/tickets/move-plan.csv`.

**Out of scope:**
- Broad cleanup not named by the taste audit.
- Behavior redesign beyond preserving existing semantics while fixing silent defects.

**Acceptance criteria:**
- Cost tracking receives `project_id` at every audited `send_message` call site.
- `backend/scripts/verify_project_id_coverage.py` exists, handles aliased imports and the `_call_claude` wrapper case, and reports `0 omissions`.
- Chat listing no longer performs the audited N+1 query pattern.
- The audited broad exception handlers no longer silently hide unexpected failures.
- Filesystem path construction follows existing path policy.
- No audited `from __future__ import annotations` instance remains unless repo guidance is explicitly changed.
- Missing `project_id` behavior is protected by the new focused cost-tracking test or equivalent.

**Verification:**

- Run `python backend/scripts/verify_project_id_coverage.py`; expect `0 omissions`.

**Traceability:** Covers taste-audit Phase 1 correctness findings before migration starts.
