# NBB-002 - Central Policy, Safety, and Shared Boundaries

## Goal

Make the high-risk contracts explicit before domain migrations begin. Epic 002 is not a serialized mega-block; each downstream stream unlocks from the tasks it actually needs.

## Outcome

Auth, permissions, raw-code analysis risk, Supabase contracts, prompt/tool assets, runtime edges, data stores, and background tasks all have owners and guardrails before major code movement.

<a id="nbb-201"></a>
### NBB-201 - Consolidate auth identity, token validation, query-token policy, and project access

**Goal:** Create one canonical identity and project-access path.

**Primary write scope:**
- `backend/app/auth/`
- `backend/app/api/auth/`
- `backend/app/utils/auth_middleware.py`
- `backend/app/services/auth/`
- route files that import auth middleware/guards

**Current repo cues:**
- `api/__init__.py` validates tokens through `utils/auth_middleware.py`.
- `services/auth/rbac.py` has a separate `RequestIdentity` path.
- Query-param tokens exist for browser media routes.
- Data services contain default-user fallback behavior.

**Scope:**
- Create one canonical `RequestIdentity` at `backend/app/auth/identity.py`.
- Create one canonical project-access check at `backend/app/auth/access.py`.
- Remove or shim duplicate token validation paths.
- Move `utils/auth_middleware.py` to `backend/app/api/auth/middleware.py`; this is API-boundary middleware because its importers are route-layer code.
- Move `services/auth/rbac.py` and `services/auth/permissions.py` under `backend/app/auth/`.
- Allow query-param token auth only for browser-loaded media/file/embed routes that cannot attach headers. JSON API routes, create/update/delete routes, and normal fetch routes must use `Authorization: Bearer`.
- Move policy decisions out of data services.

**Decision map:**

| Current code/concept | Target owner under `backend/app/` | Locked decision |
|---|---|---|
| `services/auth/rbac.py:RequestIdentity` | `auth/identity.py` | Single identity dataclass and resolver. |
| Project access checks | `auth/access.py` | Single project-access path used by routes and services. |
| `utils/auth_middleware.py` | `api/auth/middleware.py` | API middleware; not a generic utility and not domain business logic. |
| `services/auth/permissions.py` | `auth/permissions.py` | Permission policy and constants. |
| `services/auth/rbac.py` decorators | `auth/guards.py` | Route/service guard decorators built on canonical identity/access. |
| Default-user fallback | `auth/identity.py` only | Allowed only when `NOOBBOOK_AUTH_REQUIRED=false`. |
| Query-token fallback | `api/auth/middleware.py` allowlist | Only browser asset/media/embed routes that cannot send headers. |

**Mechanical vs. semantic rows:**

Mechanical rows — execute via refactory:

| Row | Mode | Tool |
|---|---|---|
| `utils/auth_middleware.py` → `api/auth/middleware.py` | `python_module_move` | `refactory:move_module` |
| `services/auth/permissions.py` → `auth/permissions.py` | `python_module_move` | `refactory:move_module` |
| `services/auth/rbac.py::require_admin` → `auth/guards.py::require_admin` | `python_symbol_move` | `refactory:move_symbol` |
| `services/auth/rbac.py::require_auth` → `auth/guards.py::require_auth` | `python_symbol_move` | `refactory:move_symbol` |
| `services/auth/rbac.py::require_permission` → `auth/guards.py::require_permission` | `python_symbol_move` | `refactory:move_symbol` |

Semantic rows — manual, do not automate beyond the initial move:
- Creating `auth/identity.py` from `RequestIdentity` and the scattered resolver code.
- Creating `auth/access.py` from the scattered project-access checks.
- Default-user fallback gating on `NOOBBOOK_AUTH_REQUIRED`.
- Query-token fallback allowlist behavior.

Mechanical rows land first via refactory, appending rows to `docs/tickets/move-plan.csv`. Semantic rows follow in separate commits within this ticket; `NBB-107` auth tests must stay green at each step.

**Out of scope:**
- Login UX redesign.
- New role model beyond what `NBB-202A` requires.

**Acceptance criteria:**
- There is one identity resolver and one project-access check.
- Query-param token behavior is documented and narrowly allowed.
- Default-user fallback is explicit dev/single-user behavior gated by `NOOBBOOK_AUTH_REQUIRED=false`.

**Verification:**
- Baseline auth tests from `NBB-107` remain green.

**Traceability:** Covers auth consolidation underspecification.

### NBB-202A - Make permissions explicit and fail-closed outside dev/single-user mode

**Goal:** Stop unknown permissions and DB failures from opening production access.

**Primary write scope:**
- `backend/app/auth/permissions.py`
- `backend/app/auth/guards.py`
- backend auth/permission tests
- frontend permission docs only when needed to mark UI behavior as non-authoritative

**Scope:**
- Define typed permission categories/items.
- Unknown categories or items must not default to allow outside dev/single-user mode.
- Permission DB failures must not return all-enabled defaults in auth-required production mode.
- Frontend fail-open behavior must be documented as UI-only and not trusted for backend authorization.

**Decision map:**

| Scenario | Required behavior |
|---|---|
| `NOOBBOOK_AUTH_REQUIRED=false` | Dev/single-user mode. Existing default-user behavior may remain, but must be explicit in code and tests. |
| `NOOBBOOK_AUTH_REQUIRED=true` or unset | Auth-required mode. Permission checks fail closed. |
| Unknown permission category | Deny with `PermissionDenied`/403; do not return allow. |
| Unknown item in known category | Deny with `PermissionDenied`/403; do not return allow. |
| DB failure while reading permissions in auth-required mode | Deny with logged warning; do not return all-enabled defaults. |
| `users.permissions is NULL` after a successful DB read | Known default permissions may apply. Unknown categories/items still deny. |
| Frontend `PermissionsContext` fail-open | UI-only behavior; backend ignores frontend-asserted permissions. |

**Out of scope:**
- Full permissions UI redesign.
- Per-customer policy customization.

**Acceptance criteria:**
- Permission behavior is fail-closed unless `NOOBBOOK_AUTH_REQUIRED=false`.
- Tests cover unknown categories and DB failure behavior.

**Verification:**
- Auth/permissions tests from `NBB-107` are updated and green.

**Traceability:** Covers fail-open permissions and knowledge-base tool gating.

### NBB-206 - Define providers and connectors boundaries and dependency-direction rules

**Goal:** Define external edge ownership without stealing domain-owned operations.

**Primary write scope:**
- `backend/app/providers/`
- `backend/app/connectors/`
- backend package charters
- structure/dependency docs

**Scope:**
- Create `providers/` and `connectors/` skeletons and boundary rules.
- `providers/` owns low-level external API clients, SDK wrappers, auth primitives, storage adapters, model-provider helpers, and runtime IO adapters.
- `connectors/<name>/` owns product-level configured external capabilities: user-visible connector stores, permission-gated tool schemas, chat-invokable connector adapters, formatting policies, and project/user integration state.
- File-format operations remain domain-owned when they express source or studio behavior.
- Explicitly reject generic `platform/files/` or `providers/files/` as the default home.
- Define dependency direction:
  - domains may depend on connector public surfaces
  - connectors may depend on provider clients
  - domains may depend on providers directly only for provider-neutral runtime/IO primitives
  - providers must not depend on connectors or domains
  - connectors must not reach into domain internals

**Out of scope:**
- Bulk migration of every provider client or connector.
- Domain orchestration.

**Acceptance criteria:**
- Domains know when to use a connector versus a provider.
- Providers contain no product orchestration.
- Connectors contain no raw SDK sprawl that belongs in providers.
- File adapters follow the explicit `NBB-401` decision map.

**Verification:**
- Wording check: no unqualified `platform/files/` or `providers/files/`.
- Verify `NBB-207B`, `NBB-207C`, `NBB-209A` through `NBB-209E`, and `NBB-705A` through `NBB-705E` targets are consistent with the providers/connectors split.

**Traceability:** Covers external-edge naming ambiguity and platform/file adapter contradiction.
