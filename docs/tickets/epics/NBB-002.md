# NBB-002 - Central Policy, Safety, and Shared Boundaries

## Goal

Make the high-risk contracts explicit before domain migrations begin. Epic 002 is not a serialized mega-block; each downstream stream unlocks from the tasks it actually needs.

## Outcome

Auth, permissions, raw-code analysis risk, Supabase contracts, prompt/tool assets, runtime edges, data stores, and background tasks all have owners and guardrails before major code movement.

## Child Tickets

<a id="nbb-201"></a>
### NBB-201 - Consolidate auth identity, token validation, query-token policy, and project access

**Goal:** Create one canonical identity and project-access path.

**Current repo cues:**
- `api/__init__.py` validates tokens through `utils/auth_middleware.py`.
- `services/auth/rbac.py` has a separate `RequestIdentity` path.
- Query-param tokens exist for browser media routes.
- Data services contain default-user fallback behavior.

**Scope:**
- Define one canonical `RequestIdentity`.
- Define one canonical project-access check.
- Remove or shim duplicate token validation paths.
- Decide where query-param token auth is allowed, limited to routes that truly cannot send headers.
- Move policy decisions out of data services.

**Out of scope:**
- Login UX redesign.
- New role model beyond what `NBB-202` requires.

**Acceptance criteria:**
- There is one identity resolver and one project-access check.
- Query-param token behavior is documented and narrowly allowed.
- Default-user fallback is explicit dev/single-user behavior.

**Verification:**
- Baseline auth tests from `NBB-107` remain green.

**Traceability:** Covers auth consolidation underspecification.

<a id="nbb-202"></a>
### NBB-202 - Make permissions explicit and fail-closed outside dev/single-user mode

**Goal:** Stop unknown permissions and DB failures from opening production access.

**Scope:**
- Define typed permission categories/items.
- Unknown categories or items must not default to allow outside dev/single-user mode.
- Permission DB failures must not return all-enabled defaults in auth-required production mode.
- Tool availability, including Notion/Jira/Mixpanel/MCP, must be filtered through the same policy path.
- Frontend fail-open behavior must be documented as UI-only and not trusted for backend authorization.

**Out of scope:**
- Full permissions UI redesign.
- Per-customer policy customization.

**Acceptance criteria:**
- Permission behavior is fail-closed outside explicit dev/single-user mode.
- All chat-invoked integrations go through the same permission path.
- Tests cover unknown categories and DB failure behavior.

**Verification:**
- Auth/permissions tests from `NBB-107` are updated and green.

**Traceability:** Covers fail-open permissions and knowledge-base tool gating.

<a id="nbb-203"></a>
### NBB-203 - Disable or flag off unsafe raw-code analysis outside dev/single-user mode before moving analysis code

**Goal:** Quarantine active `exec(code, ...)` risk before source analysis is reorganized.

**Current repo cues:**
- `analysis_executor.py` executes model-written Python with `exec(code, exec_globals)`.
- `run_analysis.json` requires a free-form `code` string.
- `csv_analyzer_agent.py` says it writes and executes pandas code.

**Scope:**
- Pre-migration mitigation only.
- Disable or feature-flag raw-code analysis off outside explicit dev/single-user mode.
- Make the disabled state obvious to users and logs.
- Document the temporary status and link permanent replacement to `NBB-404`.

**Out of scope:**
- Hard sandboxing.
- Declarative analysis redesign.

**Acceptance criteria:**
- Production/auth-required mode does not execute model-written Python.
- Dev/single-user override is explicit and named.
- Source-analysis migration cannot start before this mitigation lands.

**Verification:**
- Test that raw-code analysis is unavailable outside the allowed mode.

**Traceability:** Covers unsafe CSV analysis timing.

**Acceptance criteria:**
- Migrations and storage contracts have an owner and validation checklist.
- Domain store tickets know which schema/RLS contracts they touch.
- No one can move data stores without checking RLS/storage impact.

**Verification:**
- Review migration contract inventory.

**Traceability:** Covers Supabase/RLS/storage ownership.

### NBB-206 - Create platform skeleton and dependency-direction rules

**Goal:** Define external edge ownership without stealing domain-owned operations.

**Scope:**
- Create `platform` skeleton and adapter rules.
- Vendor SDK clients belong under platform.
- File-format operations remain domain-owned when they express source or studio behavior.
- Explicitly reject generic `platform/files/` as the default home.

**Out of scope:**
- Bulk migration of every vendor adapter.

**Acceptance criteria:**
- Domains know how to depend on platform.
- Platform has no product orchestration.
- File adapters follow the audit ownership map.

**Verification:**
- Wording check: no unqualified `platform/files/`.

**Traceability:** Covers platform/file adapter contradiction.

### NBB-207 - Define prompt and tool-schema ownership and loader compatibility

**Goal:** Allow prompt/tool JSON colocation without breaking path-based loaders.

**Scope:**
- Define where domain-owned prompt JSON and tool JSON live after migration.
- Add loader registry/shim strategy before any prompt/tool JSON move.
- Split current `chat_tools/` by owner: source search, memory, studio signal, CSV/database/Freshdesk, MCP, Jira, Notion, Mixpanel.
- Keep shared schema loaders outside feature buckets.

**Out of scope:**
- Prompt behavior redesign.
- Tool schema semantics redesign.

**Acceptance criteria:**
- Loader registry/shims land before any domain migration moves prompt JSON or tool JSON.
- Old `services/tools` and `data/prompts` are deprecated only after compatibility exists.
- Every moved JSON asset has an owning domain or integration.

**Verification:**
- Loader tests cover both old and new locations during migration.

**Traceability:** Covers prompt/tool asset and loader-risk findings.

### NBB-209 - Split data_services into domain-owned stores, landable in partial slices

**Goal:** Move CRUD stores to owning domains instead of preserving a generic data layer.

**Scope:**
- Split stores by family: chat/message, project/user, source-adjacent, settings/database/MCP, and brand.
- Each partial slice may land independently when downstream work needs it.
- Store names should be local to their domain, for example `chat/store.py`, not another `ChatService`.
- Keep Supabase/RLS implications linked to `NBB-204`.

**Out of scope:**
- Data model redesign.
- RLS rewrite unless required by store movement.

**Acceptance criteria:**
- Partial landing is allowed by store family.
- No domain waits for unrelated store moves.
- Generic `data_services` no longer accepts new work.

**Verification:**
- Check imports for each migrated store family.

**Traceability:** Covers `data_services/` ownership and unlock-matrix concerns.
