# NBB-003 - Chat Domain Migration

## Goal

Move chat to an owned domain with explicit contracts, clear policy boundaries, and domain-owned tools.

## Unlock Requirements

Chat may start after `NBB-106`, `NBB-107`, `NBB-109`, `NBB-201`, `NBB-202A`, `NBB-205`, `NBB-206`, `NBB-207A`, `NBB-207C`, and `NBB-209A`.

<a id="nbb-301"></a>
### NBB-301 - Create chat public surface and lock chat contracts

**Goal:** Create the chat import surface and preserve streaming, citation, and tool contracts.

**Primary write scope:**
- `backend/app/chat/`
- `backend/app/api/chats/`
- `backend/app/api/messages/`
- `backend/app/services/chat_services/`
- chat route smoke tests

**Scope:**
- Create `backend/app/chat` public surface with these committed exports:
  - `chat.send(project_id, chat_id, message, identity) -> ChatResponse`
  - `chat.stream(project_id, chat_id, message, identity) -> Iterator[ChatEvent]`
  - `chat.tools` for chat-owned tool policy/registry surface populated through `NBB-207A` and `NBB-207C`
  - `chat.store` for `ChatStore` and `MessageStore` re-exports needed by API-layer code
  - `chat.schemas` for `ChatResponse`, `ChatEvent`, and chat contract shapes from `NBB-205`
- Keep `backend/app/api` routes as transport adapters.
- Lock chat streaming, citation marker/lookup, and chat tool invocation/result contracts through `NBB-205`.
- Add compatibility shims only when needed for staged migration.
- Shared loop internals called by both `chat.send` and `chat.stream` live in `chat/loop.py`.

**Out of scope:**
- Moving route modules.
- Splitting `main_chat_service.py` internals; that is `NBB-302`.

**Acceptance criteria:**
- New imports can target the chat public surface.
- Route files call `chat.send` or `chat.stream` instead of legacy internals.
- Contract tests from baseline route/auth work still pass.

**Verification:** Run backend route smoke tests.

**Traceability:** Covers chat route/domain ownership and citation contract findings.

### NBB-302 - Split main_chat_service.py by real policy and state boundaries

**Goal:** Reduce the main chat service without creating shallow permanent wrappers.

**Primary write scope:**
- `backend/app/services/chat_services/main_chat_service.py`
- `backend/app/chat/context.py`
- `backend/app/chat/tool/policy.py`
- `backend/app/chat/loop.py`
- `backend/app/chat/stream.py`
- `backend/app/chat/persistence.py`
- `backend/app/chat/memory/`
- `backend/app/chat/naming.py`

**Scope:**
- Split by this target layout:
  - prompt/context assembly -> `chat/context.py`
  - tool availability and permission policy -> `chat/tool/policy.py`
  - shared agentic loop and tool-use/tool-result pairing -> `chat/loop.py`
  - streaming mechanics -> `chat/stream.py`
  - persistence coordination -> `chat/persistence.py`
  - memory invocation surface -> `chat/memory/`
  - background chat naming -> `chat/naming.py`
- Prefer locally meaningful names over `service.py`, `orchestrator.py`, or wrapper-only modules.
- `chat/loop.py` keeps class form as `ChatLoop` because the current `MainChatService` owns real state and policy: lazy tool definitions, iteration config, and the core agentic loop. Do not convert this loop to module functions; do not keep names `MainChatService` or `ChatService`.
- Temporary facade/shim is allowed only with deletion tracked by `NBB-706`.

**Out of scope:**
- Changing chat user behavior.
- Adding new tools.

**Acceptance criteria:**
- Each extracted module owns a real policy, state, IO, or contract boundary.
- No permanent pass-through wrapper is introduced.
- `main_chat_service.py` is removed or reduced to a scheduled temporary shim.

**Verification:** Run chat route smoke tests and later `NBB-701` chat/tool contract tests.

**Traceability:** Covers shallow-wrapper and naming concerns.

### NBB-303 - Assign chat-invoked tools to owning domains and connectors

**Goal:** Keep chat as orchestrator while domains own executable adapters.

**Primary write scope:**
- `backend/app/chat/`
- `backend/app/sources/`
- `backend/app/studio/`
- `backend/app/connectors/`
- `backend/app/auth/`

**Scope:**
- Chat owns tool selection/routing policy, not every tool implementation.
- Source search belongs to sources.
- Memory storage and merge behavior belong to domain-owned stores from `NBB-209A`; chat may invoke only the public memory surface governed by `NBB-202A`, `NBB-202B`, and `NBB-205`.
- Studio signal tools belong to studio.
- CSV/database/Freshdesk/deep-research tools belong to sources/analysis.
- Jira/Notion/Mixpanel/MCP tools belong to their connector/domain owners.
- Anthropic/Claude parsing and cost tracking belong under provider ownership, not chat.
- Chat tool routing consults `ToolCapabilityPolicy` from `NBB-202B`; chat must not expose a Claude-visible tool that lacks a capability entry.
- Migrated executable tool entrypoints must not keep `execute_tool`. Multi-tool routers use `dispatch(...)`; single-action tools use a domain verb.
- Standalone executable targets:
  - `source_search_executor.py::execute` -> source search public surface `search(...)`.
  - `memory_executor.py::execute` -> `chat/memory` public surface `store(...)`.
  - `studio_signal_executor.py::execute` -> `studio/signal.py::emit(...)`.
  - `web_agent_executor.py::execute_tool` -> `sources/link/tools::dispatch(...)`.
- Post-condition for migrated chat/tool surfaces: `rg 'def execute_tool\\(|\\.execute_tool\\(' backend/app/chat backend/app/sources backend/app/studio backend/app/connectors` returns zero outside historical docs/tests.

**Out of scope:**
- Moving prompt/tool JSON before `NBB-207A` lands.
- Changing tool behavior.

**Acceptance criteria:**
- Tool ownership map exists and matches prompt/tool loader plan.
- Chat does not swallow connector or analysis domains.
- Permission filtering uses `NBB-202A` and `NBB-202B`.

**Verification:** Tool contract tests in `NBB-701`.

**Traceability:** Covers chat tool ownership and Claude parsing ownership findings.

### NBB-304 - Resolve chat naming collisions

**Goal:** Prevent old `ChatService` style names from surviving in new paths.

**Primary write scope:**
- `backend/app/services/chat_services/main_chat_service.py`
- `backend/app/services/data_services/chat_service.py`
- `backend/app/chat/loop.py`
- `backend/app/chat/store.py`
- chat import/call sites

**Scope:**

`NBB-209A` performs the `ChatService` → `ChatStore` rename as part of the chat/message store move via `refactory:rename_symbol`. `NBB-304` verifies that rename landed cleanly and removes leftover `MainChatService` shims. `NBB-304` does not own a second rename pass.

- Run `mcp__refactory__validate_imports` on `backend/app` and confirm no broken references remain.
- Verify no chat class survives as `ChatService` or `MainChatService` in the migrated tree.
- Remove `main_chat_service.py` if it persists as a forwarding-only shim after `NBB-302`'s split lands.
- Confirm `chat/store.py` owns chat container persistence and `chat/message/store.py` owns message persistence.
- Forbidden class names in the migrated tree: `ChatService`, `MainChatService`.

**Out of scope:**
- Public API naming changes visible to clients.

**Acceptance criteria:**
- No two migrated chat classes/modules read as the same full path/symbol concept.
- Store and orchestration responsibilities are visibly distinct.
- `rg "class (ChatService|MainChatService)\\b" backend/app` returns zero after this ticket lands.

**Verification:** Search migrated code for forbidden class names and legacy imports.

**Traceability:** Covers chat naming collision concern.
