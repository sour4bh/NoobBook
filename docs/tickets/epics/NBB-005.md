# NBB-005 - Studio Domain Migration

## Goal

Move studio into category-owned slices while preserving service, job, tool, and run layers where they exist.

## Unlock Requirements

Studio taxonomy work may start after `NBB-104` and `NBB-207C`. Final registry and implementation work wait for route smokes, contracts, provider/connector boundaries, prompt/tool ownership, and background ownership through the exact `tickets.csv` dependencies.

<a id="nbb-501a"></a>
### NBB-501A - Finalize canonical studio taxonomy

**Goal:** Lock studio categories and item names before registry mapping.

**Primary write scope:**
- `docs/tickets/epics/NBB-005.md`
- `backend/app/studio/` taxonomy docs
- studio tool-schema ownership references from `NBB-207C`

**Scope:**
- Inventory all studio routes/items.
- Finalize canonical category and item directory names.
- The taxonomy must use only the final category and item names below.

**Decision map:**

| Category | Target item paths under `backend/app/studio/` | Current route/file concepts |
|---|---|---|
| Documents | `documents/blog/`, `documents/business_report/`, `documents/prd/`, `documents/presentation/` | Blog, business report, PRD, presentation. |
| Marketing | `marketing/ad/`, `marketing/email/`, `marketing/infographic/`, `marketing/strategy/`, `marketing/social_post/` | Ads, email templates, infographics, marketing strategies, social posts. |
| Design | `design/component/`, `design/flow_diagram/`, `design/logo/`, `design/website/`, `design/wireframe/` | Components, flow diagrams, logo support, websites, wireframes. |
| Learning | `learning/flash_card/`, `learning/mind_map/`, `learning/quiz/` | Flash cards, mind maps, quizzes. |
| Media | `media/audio/`, `media/video/` | Audio overview and video generation. |

- Category directories are plural domain groups; item directories are singular local concepts.
- Audio and video are separate item slices under `media/`; there is no combined media item.
- Logo generation is design-owned at `studio/design/logo/`; brand asset/config storage remains brand-owned.

**Out of scope:**
- Moving implementations.
- Mapping every current service/job/tool/prompt/test path; that is `NBB-501B`.

**Acceptance criteria:**
- Every studio item has one target category/item path.
- Blog, business reports, flow diagrams, and other ambiguous items have explicit ownership.
- New studio items know where to land.

**Verification:** Compare taxonomy to `backend/app/api/studio`, `studio_services`, `tool_executors`, tools, and prompts.

**Traceability:** Covers studio taxonomy ambiguity.

<a id="nbb-502"></a>
### NBB-502 - Map studio service/job/tool/run layers and status/progress contracts

**Goal:** Document the five-file item pattern, four runtime responsibilities plus a local schema file, before moving individual studio features.

**Primary write scope:**
- `backend/app/studio/`
- `backend/app/services/studio_services/`
- `backend/app/services/studio_services/jobs/`
- `backend/app/services/tool_executors/`
- studio contract docs/tests

**Scope:**
- For every studio item, map route adapter, domain service, job/background layer, sync tool dispatch, background run/agent orchestration, prompt/tool JSON, output storage, and tests.
- Preserve the distinction between sync tool dispatch and background orchestration.
- Tie studio job status/progress/result contracts to `NBB-205` and background ownership to `NBB-210`.
- `studio/export/` is a sanctioned studio capability group for cross-item export/render/screenshot operations. It is not a `shared/` bucket and must not collect generic helpers.

**Decision map:**

Each item from `NBB-501A` and the registry from `NBB-501B` uses these files inside its item directory:

| Layer | Canonical file | Replaces |
|---|---|---|
| Core operation | `<item>/<verb>.py` | `<item>_service.py` |
| Background job wiring | `<item>/job.py` | `studio_services/jobs/<item>_jobs.py` |
| Sync tool dispatch | `<item>/tool.py` | `<item>_tool_executor.py` |
| Background orchestration | `<item>/run.py` | `<item>_agent_executor.py` |
| Wire shapes | `<item>/schema.py` | New local contract file tied to `NBB-205`. |

Core operation filenames are fixed by item:

| Item path | Core operation file |
|---|---|
| `documents/blog/` | `write.py` |
| `documents/business_report/` | `write.py` |
| `documents/prd/` | `write.py` |
| `documents/presentation/` | `compose.py` |
| `marketing/ad/` | `create.py` |
| `marketing/email/` | `write.py` |
| `marketing/infographic/` | `create.py` |
| `marketing/strategy/` | `plan.py` |
| `marketing/social_post/` | `write.py` |
| `design/component/` | `build.py` |
| `design/flow_diagram/` | `build.py` |
| `design/logo/` | `create.py` |
| `design/website/` | `build.py` |
| `design/wireframe/` | `draw.py` |
| `learning/flash_card/` | `create.py` |
| `learning/mind_map/` | `build.py` |
| `learning/quiz/` | `create.py` |
| `media/audio/` | `generate.py` |
| `media/video/` | `generate.py` |

- Do not merge `job.py` into `run.py`.
- Do not preserve `<item>_service.py`, `<item>_tool_executor.py`, `<item>_agent_executor.py`, or `jobs/<item>_jobs.py` as renamed files in the new item path.
- Each item maps all possible layers, but creates files only for layers that exist in that item's behavior.
- If an item has no background run layer, do not create `run.py`.
- If an item has no sync tool dispatch layer, do not create `tool.py`.
- If an item has no job-wiring layer, do not create `job.py`.
- If an item has no item-specific request/result/wire shapes, do not create `schema.py` just to re-export shared schemas.
- The registry records `none` for absent layers and flags that absence as intentional.

**Out of scope:**
- Merging layers for convenience.

**Acceptance criteria:**
- Each studio item has a core operation/job/tool/run/schema map, including explicit `none` entries for absent layers.
- Migrators cannot silently merge jobs into executors.
- Migrators cannot create placeholder files only to satisfy the pattern.
- Shared studio behavior has a chartered home.

**Verification:** Review map against `studio_services/jobs`, `tool_executors`, and studio routes.

**Traceability:** Covers studio jobs and executor-pair invariant.

### NBB-504 - Migrate document items

**Goal:** Move document generators into registry-approved document homes.

**Primary write scope:**
- `backend/app/studio/documents/`
- legacy document studio services/jobs/tools
- document prompt/tool JSON after `NBB-207A`
- document studio tests

**Scope:**
- Migrate document-category items from the registry.
- Co-locate item-specific core operation, job, tool, run, schema, prompt, tool-schema, and tests with each item when those layers exist.
- Use shared code only through chartered `shared/` homes.
- Use `refactory:move_module` per legacy service file and `refactory:rename_symbol` per class, with `NBB-502`'s verb table as the target filename (for example `blog_service.py` → `documents/blog/write.py`). Append one row per operation to `docs/tickets/move-plan.csv` with `ticket=NBB-504`.

**Out of scope:**
- Marketing, design, learning, or media items.

**Acceptance criteria:**
- Document items no longer depend on generic executor ownership.
- Prompt JSON movement follows `NBB-207A` and `NBB-207B`; tool JSON movement follows `NBB-207A` and `NBB-207C`.
- Route adapters use item public surfaces.

**Verification:** Studio smoke tests and later `NBB-703`.

**Traceability:** Covers studio document migration.
