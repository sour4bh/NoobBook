# NBB-009 - Deferred Closure and Runtime Hardening

<a id="nbb-009"></a>
## Epic Goal

Close the deferred post-migration items that are now small enough to finish.
The first hardening wave (`NBB-901` through `NBB-909`) handled config/runtime
cleanup, auth transport parity, OAuth callback state validation, project access
semantics, API transport-boundary enforcement, safe CSV analysis, and frontend
smoke coverage.

The second hardening wave (`NBB-910` through `NBB-918`) turns the remaining
deferred `D-003` and `D-005` risks into executable tickets: signup/admin
separation, scoped browser asset access, service-role ownership checks,
provider/connector permission gates, storage isolation, OAuth replay hardening,
backend-owned DTOs, frontend runtime parsers, and final deferred-state
reconciliation.

## Execution Notes

- Run mostly serially because config, app bootstrap, auth middleware, and
  source-analysis contracts share import and behavior surfaces.
- `NBB-908` may run after `NBB-901`; it is frontend-only.
- No compatibility shims are allowed for old config module names.
- HTTP route paths stay stable.
- `NBB-910` through `NBB-918` should run mostly serially. `NBB-914` may run in
  parallel with `NBB-913`/`NBB-915` after `NBB-912` if storage and OAuth files
  are confirmed disjoint.
- Public signup may create a workspace owner only inside a workspace-scoped
  ownership model. It must never implicitly create a global admin. Global admin
  bootstrap remains env, CLI, or invite only.

### NBB-910 - Split signup ownership from global admin bootstrap

**Goal:** Make public signup safe for hosted use: the first public signup can
own its workspace, but public signup must not create a global admin.

**Primary write scope:**
- `backend/app/auth/`
- signup/auth routes and tests
- frontend auth copy only if required by changed semantics
- auth/deployment docs that describe bootstrap behavior

**Scope:**
- Remove any "first public signup becomes global admin" behavior.
- Keep global admin creation on explicit env, CLI, or invite/bootstrap paths.
- If the current tree has no workspace/org model, public signup creates a
  normal user only and this ticket documents workspace-owner creation as a
  future product model dependency.
- If a workspace/org model is present or added in this ticket, public signup may
  create a workspace and owner membership scoped to that workspace.
- Add regression tests for first signup, later signup, explicit global-admin
  bootstrap, and permission claims exposed to the frontend.

**Out of scope:** Full team/workspace membership redesign beyond the minimal
owner/global-admin separation needed here.

**Acceptance criteria:** No unauthenticated public signup path grants global
admin. Explicit bootstrap still creates or recognizes a global admin through a
non-public path.

**Verification:** Auth/signup tests, permission tests, full backend pytest,
pyright, architecture checks, and `dag.py --check`.
