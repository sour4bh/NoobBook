# NBB-010 - Workspace Membership and Project Sharing

<a id="nbb-010"></a>
## Epic Goal

Resolve the remaining `D-005` product/API contract gap by adding real
workspace and project membership semantics. Public signup creates a personal
workspace where the signing-up user is workspace `owner`. Users join other
workspaces only through signed invites. Projects are private by default and are
visible only through explicit project membership.

Global admin remains an instance/bootstrap role for operators. It is not the
normal team-owner concept, and public signup must never create global admins.

## Role Model

| Scope | Roles | Meaning |
|---|---|---|
| Global | `admin`, `user` | Instance/bootstrap administration only. Existing `users.role` remains the source for this scope and is surfaced as `global_role` in new contracts. |
| Workspace | `owner`, `admin`, `member` | Team/settings scope. `owner` and `admin` can manage workspace team, settings, provider keys, models, and brand. |
| Project | `owner`, `editor`, `viewer` | Private project access. Workspace membership alone does not grant project visibility. |

## Execution Notes

- Run serially. Auth/session contracts, project guards, settings ownership, and
  frontend state all share surfaces.
- No compatibility shim is required for the old global Team settings UI.
- Existing HTTP project detail/chat/source/studio paths stay stable; their
  guards switch to project membership.
- Provider secrets become workspace-scoped and encrypted at rest. `.env`
  provider keys remain documented only as bootstrap/default fallback values.
- Storage object ownership must become workspace/project-aware so shared
  project members can access assets through the same membership model.

<a id="nbb-1004"></a>
### NBB-1004 - Enforce private project membership access

**Goal:** Switch project access from owner-only `projects.user_id` checks to
explicit `project_members` roles.

**Primary write scope:**
- `backend/app/projects/`
- `backend/app/auth/access.py`
- `backend/app/__init__.py`
- `backend/app/api/projects/`
- project/auth tests

**Scope:**
- `GET /projects?workspace_id=...` lists projects where the caller has project
  membership in that workspace.
- `POST /projects` requires `workspace_id` and creates owner project
  membership for the creator.
- Reads allow `owner`, `editor`, and `viewer`.
- Mutations allow `owner` and `editor` where appropriate.
- Delete/share ownership-only behavior is reserved for `owner`.

**Out of scope:** Team/invite endpoints and frontend project sharing UI.

**Acceptance criteria:** A workspace member without project membership cannot
see or open the project. Viewer can read but not mutate. Editor can mutate
content but cannot delete/share. Owner can manage the project.

**Verification:** Guard tests, route smoke tests, full backend pytest,
architecture checks, pyright, and `dag.py --check`.
