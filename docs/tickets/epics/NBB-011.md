# NBB-011 - Typed Agent Runtime and Multi-Provider LLM Layer

<a id="nbb-011"></a>
## Epic Goal

Replace the Anthropic-shaped AI integration with a typed provider-neutral
runtime at `backend/app/agents/runtime`. The app remains the owner of message
history, tool dispatch, citations, memory, cost tracking, project/workspace
state, and persistence. Provider adapters translate between that runtime
contract and external APIs.

Anthropic Messages remains the default provider. OpenAI Responses becomes a real
second provider after the runtime and Anthropic adapter are stable. OpenAI
Agents SDK and Claude Agent SDK stay out of core chat in this epic.

## Runtime Boundary

`backend/app/agents/runtime` owns only:

- typed run, message, event, tool, usage, provider-state, and structured-output
  contracts;
- shared local-tool execution mechanics;
- strict schema generation from Pydantic tool models;
- fake provider harnesses used by contract tests.

Domain tools and prompts remain under their owning domains or connectors.
Provider SDK clients and protocol translation remain under `backend/app/providers`.

## Decision Maps

### Runtime ownership

| Concept | Owner | Notes |
|---|---|---|
| Neutral run/message/event contracts | `backend/app/agents/runtime` | App-owned contract consumed by chat, sources, studio, and provider adapters. |
| Local tool input/output contracts | Owning domain or connector | Exposed as Pydantic-backed `ToolSpec`s; runtime only executes compiled specs. |
| Provider protocol conversion | `backend/app/providers/{anthropic,openai}` | Adapters translate between runtime messages/events and vendor SDK request/response shapes. |
| Built-in prompt contracts | Domain-owned `prompt.py` modules | `NBB-1113` moves prompt text/config from JSON into typed Python `PromptSpec`s with Pydantic render contexts. |
| Model catalog and provider selection | `backend/app/config/model.py` | Provider-aware model metadata and category overrides; Anthropic remains default. |
| Cost usage normalization | Provider modules plus project cost contract | Runtime carries normalized usage; provider modules price provider/model records. |
| Message persistence | `backend/app/chat/message/` | Stores neutral content parts after `NBB-1106`; legacy Anthropic JSON is migration input only. |

### Legacy surface retirement

| Current surface | Retirement ticket | Replacement |
|---|---|---|
| `claude_service.send_message` calls in domains | `NBB-1106` through `NBB-1108` | Runtime runner plus selected provider adapter. |
| `app.providers.anthropic.response_parser` in domains | `NBB-1105` through `NBB-1108` | Adapter-owned parsing into `RunResult` and `RunEvent`. |
| `app.providers.anthropic.content` in domains | `NBB-1105` through `NBB-1108` | Adapter-owned message encoding and tool-result formatting. |
| `backend/app/**/tools/*.json` | `NBB-1104` | Domain-owned Python `ToolSpec`s with Pydantic schemas. |
| `backend/app/**/prompts/*_prompt.json` | `NBB-1113` | Domain-owned Python `PromptSpec`s rendered through `app.config.prompt`. |
| `ToolLoader` dict-output APIs | `NBB-1104` and `NBB-1111` | Domain tool registries returning `ToolSpec`s. |
| Claude-only model catalog and cost buckets | `NBB-1103`, `NBB-1110` | Provider-aware model metadata and usage records. |

### Provider defaults

| Purpose | Initial provider | Reason |
|---|---|---|
| Existing chat, source, and studio prompts | Anthropic Messages | Preserve current product behavior while runtime contracts land. |
| OpenAI test prompts | OpenAI Responses | Prove second-provider support without switching default workflows. |
| Server/hosted provider tools | Provider-specific `ToolSpec` variants | Do not flatten hosted tools into local function calls. |
| SDK agent frameworks | Out of core-chat scope | Agents SDKs may be future specialist runtimes, not this app-owned loop. |

## Execution Notes

- Run `NBB-1101`, `NBB-1102`, `NBB-1103`, `NBB-1104`, `NBB-1105`, and
  `NBB-1106` serially. These tickets define the shared contracts that every
  later caller uses.
- After `NBB-1106`, `NBB-1107` and `NBB-1108` may run in parallel only if their
  write scopes stay under `backend/app/sources/` and `backend/app/studio/`.
- `NBB-1109` may start after `NBB-1105`; merge it before `NBB-1110` so
  cross-provider tests can use the real OpenAI adapter.
- `NBB-1110` and `NBB-1111` are serial finalization gates. `NBB-1112` and
  `NBB-1113` are post-finalization cleanup passes for hierarchy and prompt
  contract shape discovered while reviewing the completed runtime shape.
- No domain module should branch on provider-specific response shapes after its
  migration ticket lands.
- Tool schemas become Python/Pydantic first. Generated Anthropic/OpenAI schemas
  are outputs, not checked-in source truth.
- Built-in prompt contracts become Python/Pydantic first. JSON remains valid
  only for external/user-editable data such as persisted project custom prompts,
  not for app-owned prompt behavior.
- Existing Anthropic-shaped chat JSON is a migration boundary, not a permanent
  internal compatibility contract.
- Do not add `app.services`, `app.utils`, or generic helper roots while doing
  this migration. Runtime code is a narrow orchestration contract, not a dumping
  ground for domain behavior.

### NBB-1112 - Collapse trivial echo tool bindings

**Goal:** Remove tool-binding modules whose only job is to apply `echo_input`
to a single internal tool, while preserving catalog-owned static `ToolSpec`
contracts.

**Primary write scope:**
- `backend/app/sources/{pdf,pptx,image}/`
- selected one-shot studio domains if they meet the same criteria
- `backend/scripts/verify_architecture.py`
- focused source/studio/runtime tests

**Scope:**
- Collapse only binding modules that bind one local tool, use only
  `echo_input`, have no stateful executor, no connector call, no domain result
  formatting, and no recoverable error mapping.
- Use direct `bind_local_tools(...)` in the owning workflow module for these
  exact echo-only cases. Do not add a replacement helper; a separate binding
  module is earned by domain complexity, not by visual consistency.
- Keep `tools/specs.py` in place as the catalog-owned provider-visible
  contract.
- Start with PDF, PPTX, and image extraction bindings. Then consider the
  flow-diagram, flash-card, mind-map, and quiz one-shot studio bindings if their
  `terminating_tools={...}` behavior is already represented by `ToolSpec`.
- Update the architecture verifier so it distinguishes missing bindings by
  accident from intentionally colocated single-tool echo bindings.

**Out of scope:** Binding modules that bind multiple tools, dispatch real
effects, keep per-run state, format business results, or convert failures into
`ToolOutput(is_error=True)`.

**Acceptance criteria:** Trivial binding files in the chosen scope are deleted,
non-trivial binding modules remain separate, static `tools/specs.py` contracts
remain catalog-owned, and the architecture verifier encodes the exception
without weakening the default binding check.

**Decision note:** This ticket intentionally does not make every domain use the
same file breakdown. Stateful or effectful tool families keep `binding.py`;
single terminating echo tools stay local to the caller because the path and
`ToolSpec` already carry the domain meaning.

**Verification:** Focused PDF/PPTX/image and selected studio tests, runtime tool
contract tests, `python backend/scripts/verify_architecture.py`, backend pytest
where touched, and `python docs/tickets/dag.py --check`.
